# Task Service

Реализация тестового задания для компании **Medods**.

**Автор:** Давиденко Дмитрий Николаевич.

Сервис для управления задачами с HTTP API на Go.

## Описание тестового задания

В рамках тестового задания в проект добавляется поддержка **периодических задач**.

Поддерживаемые сценарии периодичности:

- ежедневные задачи — каждый `n`-й день;
- ежемесячные задачи — на конкретное число месяца от `1` до `30`;
- задачи на конкретные даты;
- задачи на чётные или нечётные дни месяца.

## Принятые предположения

Так как исходное задание не фиксирует все детали реализации, в проекте принимаются следующие допущения:

1. Репозиторий содержит **backend API**, а не полноценный пользовательский интерфейс. Для ручной проверки используется Swagger UI.
2. Сущность `Task` рассматривается как **конкретный экземпляр задачи**, который отображается в трекере и по которому меняется статус.
3. Настройки периодичности должны храниться отдельно от конкретных экземпляров задач, так как одна периодическая задача порождает серию задач на разные даты.
4. Для корректной работы периодичности у конкретной задачи должна существовать дата и время выполнения — `scheduled_at`.
5. Периодические задачи должны материализоваться в обычные задачи на ограниченный горизонт планирования, без обязательного введения отдельного фонового планировщика.
6. Изменение правил периодичности должно влиять только на будущие невыполненные экземпляры задач. Исторические и завершённые задачи должны сохраняться.
7. Для ежемесячных задач допустимы только дни месяца от `1` до `30` — в соответствии с условием задания.
8. Экземпляры задач, созданные по правилам периодичности, создаются со статусом `new`.

## Проектные решения для реализации

Для поддержки периодичности предполагается следующее разделение ответственности:

- **Task** — конкретная задача на определённую дату;
- **Task recurrence** — правило, по которому создаются будущие экземпляры задач.

Такое разделение позволяет:

- не смешивать шаблон и фактическую задачу в одной записи;
- хранить историю уже созданных и выполненных задач;
- безопасно изменять правила генерации только для будущих экземпляров;
- избежать дублей при повторной генерации задач.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8000`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

## Swagger

Swagger UI:

```text
http://localhost:8000/swagger/
```

OpenAPI JSON:

```text
http://localhost:8000/swagger/openapi.json
```

## API

Базовый префикс API:

```text
/api/v1
```

Текущие основные маршруты:

- `POST /api/v1/tasks`
- `GET /api/v1/tasks`
- `GET /api/v1/tasks/{id}`
- `PUT /api/v1/tasks/{id}`
- `DELETE /api/v1/tasks/{id}`

## Что будет добавлено в рамках тестового задания

В рамках реализации поддержки периодических задач API будет расширено полями и/или сущностями, необходимыми для:

- хранения настроек периодичности;
- хранения даты выполнения конкретной задачи;
- генерации экземпляров задач по заданному правилу;
- безопасного обновления и удаления периодических серий без потери истории.
